# Confluence Human Lineage Page Contract

This contract implements the first slice of
`docs/adr/ADR-009-Human-Centered-Confluence-Lineage-Catalog.md`.
Database Catalog pages must also follow
`docs/adr/ADR-013-Complete-Database-Catalog-And-Object-Library-Pages.md` and
`docs/CONFLUENCE_DATABASE_CATALOG_LAYOUT.md`. Platform-grouped Database Catalog
paths must follow `docs/adr/ADR-021-Platform-Grouped-Database-Catalog.md`, and
complete Tier 2 object coverage plus schema hyperlinks must follow
`docs/adr/ADR-022-Complete-Tier2-Object-Pages-And-Schema-Hyperlinks.md`.

The goal is to make Confluence useful to humans without turning it into the
machine-readable source for the Sonic lineage skill. Azure DevOps lineage
artifacts and runtime packages remain authoritative for automated answers.

## Scope

Use this contract for human-facing Confluence pages:

- product pages
- database pages
- schema pages
- canonical object pages

Do not use this contract for AI retrieval artifacts such as object locators,
quick context pages, catalog shards, answer cards, or compact context packs.
Those artifacts should remain under a separated `AI Retrieval Artifacts` section.
Use `docs/ROVO_AI_RETRIEVAL_ARTIFACTS_CONTRACT.md` when the goal is Rovo
question answering rather than human browsing.

Do not publish user/account schemas as human browse pages. The current blocked
schema list is maintained in `docs/CONFLUENCE_FULL_REBUILD_SCOPE.md`; the
renderer must suppress those schemas from schema page generation and database
summary tables.

## Evidence Packet

Every human-facing generated page must be built from a bounded evidence packet.
The packet is produced by deterministic metadata extraction before any LLM prose
is generated.

Required common fields:

```json
{
  "page_type": "product | database | schema | object",
  "page_title": "",
  "canonical_id": "",
  "source_artifact_paths": [],
  "generated_at": "",
  "evidence_hash": "",
  "confidence": {
    "label": "",
    "score": null,
    "validated_edges": null,
    "unresolved_facts": null,
    "caveats": []
  },
  "plain_english_inputs": {
    "business_subject": "",
    "source_summary": "",
    "target_summary": "",
    "support_impact": "",
    "missing_facts": []
  }
}
```

Database and schema object rows must include deterministic link-state fields:

```json
{
  "canonical_page_path": "",
  "canonical_page_exists": false,
  "planned_in_packet": false,
  "link_status": "linked | pending | blocked",
  "link_status_reason": ""
}
```

Use `linked` only after a live publish/readback confirms the canonical human
object page exists. Use `pending` for dry-run rows and for rows whose Tier 2
object page has not been published yet. Use `blocked` only when a blocked-schema
or do-not-publish rule prevents a human object page.

## Plain-English Rules

Plain-English text may be generated by an LLM only after the evidence packet is
created.

The LLM prompt must:

- receive only the bounded evidence packet and page contract
- preserve exact technical names in code formatting
- explain what the page represents in business/support language
- label missing fields as `not surfaced in metadata`
- avoid inventing owners, reports, source systems, targets, SLAs, or business
  definitions not present in the packet
- produce concise prose that a support analyst can understand before expanding
  technical details

Preferred summary shape:

```text
<Name> represents <business subject>. It is fed by <upstream summary> and
supports <target/consumer summary>. If it is stale or wrong, <support impact>.
Start troubleshooting by checking <first concrete check>.
```

If a field is missing:

```text
The final target was not surfaced in metadata.
```

Do not write:

```text
This object handles data movement.
```

That is too generic to help a human.

## Page Templates

### Product Page

Required sections:

1. `Plain-English Summary`
2. `Business Value`
3. `Key Data And Final Targets`
4. `Lineage Summary`
5. `Support Checks`
6. `Known Gaps And Confidence`
7. `Related Pages`
8. `Technical Evidence`

Minimum support checks:

- check the top-most orchestration jobs
- check row freshness in key final targets
- check upstream source availability
- check downstream report/dashboard complaints against named target objects

### Database Page

Required sections:

1. `Plain-English Summary`
2. `At A Glance`
3. `Schemas`
4. `Priority Object Tags`
5. `Profile Coverage`
6. `Known Gaps And Confidence`
7. `Related Product Areas`
8. `Technical Evidence`

### Schema Page

Required sections:

1. `Plain-English Summary`
2. `At A Glance`
3. `Most Used Objects`
4. `Tables`
5. `Views`
6. `Procedures`
7. `Other Objects`
8. `Objects With Profile Data`
9. `Objects Needing Review`
10. `Known Gaps And Confidence`
11. `Technical Evidence`

Schema pages must be complete object-library indexes. They must list every
cataloged object in the schema after blocked-schema rules are applied, grouped
by useful sections rather than limited to high-use objects only.

Schema rows must not rely on display-name-only matching for future hyperlinks.
They must carry the canonical object page path and link status in the evidence
packet even when the rendered row remains plain text during dry run.

### Object Page

Required sections:

1. `Plain-English Summary`
2. `Business Meaning And Impact`
3. `At A Glance`
4. `Lineage Summary`
5. `Support Checks`
6. `Profile And Quality Signals`
7. `Related Pages`
8. `Technical Evidence`

Object pages should use expandable sections for:

- upstream objects
- downstream objects
- loaders and orchestrators
- columns and column usage
- procedure/package/report calls
- confidence and unresolved facts
- source artifact paths

Object pages must live under the canonical database/schema path when published
to the human catalog:

```text
Database Catalog / <Platform/Product> / <Database> / <Schema> / <Object>
```

Do not create a separate top-level `High-Value Assets` object hierarchy. Use
`high-value`, `high-use`, `product-critical`, `support-critical`, `profiled`,
`review-needed`, and `lineage-hotspot` as tags
on object rows/pages instead.

## Quality Gate

A generated human page fails validation when:

- the plain-English summary does not name a business/data subject
- the summary says there are targets or consumers but none are present in the
  evidence packet
- available upstream/downstream evidence is omitted from the first two sections
- missing source, target, report, profile, or confidence evidence is not labeled
  as missing or not surfaced
- support checks are generic and do not name a package, procedure, table, view,
  report, or source area when evidence provides one
- the page exposes secrets, raw rows, sample values, credentials, tokens, or
  sensitive connection-string values
- AI retrieval artifact pages are mixed into the primary human navigation tree
- schema pages omit cataloged objects from the complete object index
- schema pages use noncanonical generated titles such as
  `Schema - <Database>.<Schema>` under the database page
- object pages are published under `High-Value Assets` instead of the canonical
  database/schema path

Minimum dry-run review prompts:

1. Can a human answer what this page is about from the first paragraph?
2. Can support identify what to check first?
3. Are source, target, and downstream impact backed by packet evidence?
4. Are weak or missing facts labeled honestly?
5. Is the technical detail available without overwhelming the top of the page?
